# 项目盘点与范围冻结

## 1. 项目三端现状总览

### 判定口径

- `已开发`：仓库中已有真实页面或接口实现，且主路径已接真实后端或真实本地能力。
- `半开发/占位`：已有页面或接口，但仍依赖静态数据、mock、硬编码 demo 用户、参考页 iframe，或只覆盖部分链路。
- `未开发`：仓库里只有 schema、文档、菜单入口或设计草稿，没有对应业务实现。

### 总体判断

| 端 | 现状判断 | 结论 |
| --- | --- | --- |
| `backend` | 原生 `http` + Prisma + PostgreSQL 单体服务，认证、星球、内容、加入订单链路已落代码和数据库写入。 | 已有可联调主干，但“真实支付、内容访问控制、后台真实统计”仍未收口。 |
| `admin-web` | React + Vite 项目已接登录、用户侧星球列表/主页/加入预览；真正的后台菜单多数还是静态样例、占位数据或参考页 iframe。 | 不是空壳，但“管理后台”本体仍是半成品。 |
| `miniprogram` | 原生微信小程序约束基本遵守，`wx.request` / `wx.uploadFile` / `wx.login` 已接入；知识星球主链路大部分页面已存在。 | 已具备知识星球子模块的基础可跑版本，但仍混有历史本地存储页和非主线静态页。 |
| `docs` | SOP、后端/后台规划、串行提示词都已存在。 | 文档体系完整，但部分文档已落后于真实代码，后续应以代码为准。 |

### 核心闭环的真实结论

- 已经具备“能跑”的主干：登录、发现、主页、发帖、评论、点赞、我的星球。
- 当前最薄弱的是“加入”与“后台数据页”：
  - 加入链路虽然能下单、开会员、跳转主页，但支付仍是 mock。
  - 管理后台虽然菜单齐，但真实可用的数据页只有极少数，且推广数据仍是假数据。

## 6. 第一轮范围冻结

### 本轮必须做

1. 以后端为先，把登录态、星球发现/加入、星球主页、发帖、评论、点赞、我的星球这条主链路稳定下来。
2. 保留现有 `backend/src/server.js + src/services + prisma/schema.prisma` 结构，不在第一轮先做框架级重构。
3. 统一后端身份来源，逐步把依赖外部 `userId` 的接口收口为基于 `sessionToken` 的可信用户身份。
4. 给内容读接口补访问控制，至少要让“是否已加入/是否可看内容”成为后端判断，而不是只靠前端页面跳转。
5. 把 admin-web 聚焦到最小可用面：`/login`、`/group_data`、`/group/:groupId`、`/preview/:groupId`、`/promotion/data` 或 `/income` 至少要能看真实后端数据。
6. 把小程序聚焦到知识星球子模块：登录、发现、加入、主页、发帖、评论、点赞、我的星球、创建星球、调试页。
7. 加入闭环第一轮允许继续用 mock 支付做内部联调，但订单、会员开通、主页访问、内容访问这几步必须统一到同一套后端逻辑上。

### 本轮建议暂缓

1. 续期、续期优惠券、续期通知、续期折扣整块能力。
2. 优惠券后台、渠道二维码后台、分组通知后台、权限后台的完整配置能力。
3. 积分榜、星球豆、创作灵感、视频号直播、成员身份验证、导出任务等运营增强能力。

## 7. 给 02 号提示词的输入摘要

- `backend` 不是从零开始，已经有真实 service 主干：认证、星球、内容、加入订单、图片上传、seed、验流脚本都在。

- `admin-web` 当前应被视为“两层结构”：
  - 一层是已接后端的用户侧联调页：登录、星球列表、星球主页、加入预览。
  - 一层是仍未真正实现的后台页：绝大部分菜单页仍是静态数据、占位或 iframe 参考页。

  - 已接后端的核心页：登录、发现、我的星球、主页、帖子详情、评论、点赞、发帖、创建星球、调试页。

- 02 号提示词应重点审计 backend 的 4 类缺口：
  1. session 与用户身份统一。
  2. 星球/帖子/评论的读权限与会员访问控制。
  3. 加入订单与会员开通闭环中仍依赖 mock 的环节。
  4. admin-web 所需最小真实统计接口。

# 后端现状审计与缺口清单

## 前置最小结论

01 号结论在当前 headless 产物里只有 dry-run 占位，因此这里先重建一个最小前置结论：

- `backend` 目前是一个独立的 Node.js + Prisma + PostgreSQL 服务，入口集中在 `backend/src/server.js`，没有使用 Express/Koa，而是用原生 `http.createServer` 手写路由分发。证据：`backend/src/server.js:1-609`。
- 业务代码已经按 `authService / planetService / joinFlowService / contentService / wechatService` 分层，说明一期实现思路已经固定为“单体服务 + service 文件拆分”，03-04 最好沿着这个结构继续补齐，而不是先重构框架。证据：`backend/src/server.js:5-34`，`backend/src/services/*.js`。
- Prisma schema 远大于当前 service 落地范围：`schema.prisma` 一共定义了 34 个 model，但真正被 service 直接或间接落地的只覆盖认证、星球、内容、加入支付这一小段闭环。证据：`backend/prisma/schema.prisma:198-872`，`backend/src/services/*.js`。

运行时验证备注：

- 仓库已经提供 `db:start`、`db:seed`、`verify:join-flow`，说明作者预期本地可以直接做联调与验流。证据：`backend/package.json:7-15`，`backend/docker-compose.yml`，`backend/scripts/verifyJoinFlow.js:1-49`。
- 2026 年 4 月 3 日重新执行 `npm run verify:join-flow` 仍然失败，原因是本地 PostgreSQL 未启动，`localhost:5432` 不可达。因此本报告以静态代码审计为主，不把“脚本存在”误判成“当前环境已验证跑通”。证据：`backend/src/db/seedDemoData.js:35-64`，`backend/scripts/verifyJoinFlow.js:14-46`。

## schema 落地度分析

### 总览

- `schema.prisma` 共定义 34 个 model。证据：`backend/prisma/schema.prisma:198-872`。
- 其中 18 个 model 已被 service 直接通过 `prisma.xxx` / `tx.xxx` 使用；2 个 model 通过 relation 的嵌套 `create/include` 间接落地；12 个 model 只在删除脚本等工具链出现，业务接口未落地；2 个 model 目前在 backend 代码里完全未见消费。

### A. 已被真实 service 直接落地的模型

这些模型已经不是“只有表”，而是已经进入真实 API 的读写链路：

- 认证/用户：`User`、`AuthSession`。证据：`backend/src/services/authService.js:220-430`，`backend/src/services/planetService.js:41-80`。
- 星球与成员：`Group`、`GroupPermissionPolicy`、`GroupStaff`、`GroupMember`。证据：`backend/src/services/planetService.js:82-167`，`backend/src/services/joinFlowService.js:17-59`，`backend/src/services/contentService.js:661-709`。
- 内容：`Post`、`Comment`、`PostLike`、`CommentLike`。证据：`backend/src/services/contentService.js:363-1236`。
- 交易：`Order`、`PaymentRecord`、`Coupon`、`CouponClaim`、`PromotionChannel`、`OrderStatusLog`。证据：`backend/src/services/joinFlowService.js:144-556`。
- 事件/异步：`AnalyticsEvent`、`OutboxJob`。证据：`backend/src/services/joinFlowService.js:234-246`，`backend/src/services/joinFlowService.js:489-521`。

### B. 已间接落地，但接口能力明显不足的模型

- `UserProfile`：service 没有直接 `prisma.userProfile` 调用，但登录、会话、帖子作者信息都依赖 `profile` relation；这说明资料表已经真实生效。证据：`backend/src/services/authService.js:287-380`，`backend/src/services/contentService.js:106-123`。
- `RenewalSetting`：建星球时会创建，星球主页查询也会 include，但目前没有独立的续期配置接口，也没有续期下单逻辑。证据：`backend/src/services/planetService.js:124-142`，`backend/src/services/contentService.js:216-217`。

### C. schema 里有表，但业务接口几乎未落地

这一组不是“没设计”，而是典型的“不是没表，而是没接口”：

- 运营通知：`GroupNotification`、`RenewalNotificationTemplate`。
- 积分/榜单：`ScoreboardSetting`、`ScoreRule`、`MemberScoreLedger`。
- 身份会员与导出：`IdentityMemberContract`、`ExportTask`。
- 后台统计：`GroupIncomeDailyStat`、`GroupPromotionDailyStat`、`GroupMemberDailyStat`、`GroupContentDailyStat`、`GroupRenewalDailyStat`。

这些模型当前只在 `deleteGroup.js` 的依赖树清理逻辑里出现，没有对应 service、没有 HTTP 路由，也没有前端消费。证据：`backend/scripts/deleteGroup.js:19-276`，`backend/prisma/schema.prisma:563-862`。

### D. schema 里有表，backend 代码里几乎完全未使用

- `HotBoardSnapshot`
- `HotBoardItem`

当前 backend 代码里既没有 service 使用，也没有脚本使用。证据：`backend/prisma/schema.prisma:712-737`。

### 结构性判断

- 当前 schema 的设计明显超前于一期代码实现，尤其是统计、续期、通知、积分、导出等后台运营能力已经先画表了，但没有落到 service 和 route。03 号规划不应再从“补表”开始，而应从“给现有表补可被前端消费的最小接口”开始。
- 真正可复用的主干模型已经足够支撑一期：`User/AuthSession/Group/GroupMember/Post/Comment/Order/PaymentRecord/Coupon/PromotionChannel/AnalyticsEvent`。因此 03-04 更适合围绕这些表补闭环，而不是再发散扩表。

## 当前最关键缺口

当前最关键的缺口不是某一张表缺失，而是“身份闭环和内容访问闭环还没有真正成立”：

- 对外接口里仍有多处直接接受 `userId`，没有强制从 session 派生当前用户，导致预览页、星球主页、下单、会员状态都存在被冒用的空间；`contentService` 也保留了 `userId` 回退逻辑。证据：`backend/src/server.js:248-297`，`backend/src/services/contentService.js:99-139`，`backend/src/services/joinFlowService.js:145`，`backend/src/services/joinFlowService.js:558`，`backend/src/services/authService.js:597-666`。
- 内容读侧没有把“付费成员才能看内容”收紧到接口层，导致即使订单和会员关系已经能回写，付费内容仍可能被未加入用户直接读取。证据：`backend/src/services/contentService.js:202-361`，`backend/src/services/contentService.js:363-629`，`backend/src/services/contentService.js:915-945`。

如果这两个问题不先补，后面即便接上真实支付，闭环仍然是假的。

## 最建议先补的 3-5 个后端能力

1. 统一鉴权入口，把所有需要身份的接口改成“只认 session，不再信任外部传入 `userId`”。
2. 给星球主页、帖子流、帖子详情、评论列表补成员访问控制，把“预览可见”与“入圈可见”正式拆开。
3. 把加入支付从 mock callback 升级为真实微信支付链路，至少包含统一下单参数生成、支付结果回调、回调验签与幂等。
4. 把 `/api/admin/promotion/data` 改成真实聚合接口，优先从 `Order`、`PaymentRecord`、`AnalyticsEvent`、`GroupMember` 聚合，不必一开始就写整套日统计任务。
5. 补最小运营接口：优惠券列表/创建/启停、渠道码列表/创建/启停；续期与通知先只保留 schema，不在一期强推大而全。

## 给 03 号提示词的输入摘要

可直接把下面这段作为 03 的输入基线：

- 后端一期不要重构框架，继续沿用 `backend/src/server.js + src/services + prisma/schema.prisma` 现有结构。
- 一期只围绕知识星球最小闭环补能力，优先级顺序是：
  1. 登录态与身份统一。
  2. 内容读权限收口。
  3. 真实支付接入与会员开通幂等。
  4. admin 统计接口真实化。
  5. 优惠券/渠道码最小运营接口。
- 一期优先复用现有模型：`User`、`AuthSession`、`Group`、`GroupMember`、`Post`、`Comment`、`Order`、`PaymentRecord`、`Coupon`、`PromotionChannel`、`AnalyticsEvent`、`OrderStatusLog`、`OutboxJob`。
- 一期暂不处理的大块范围：续期通知、积分榜单、身份会员、导出任务、热榜抓取。
- 03 号阶段计划建议至少明确：
  1. 哪些阶段是 admin-web 前置依赖。
  2. 哪些阶段是小程序前置依赖。
  3. 哪些阶段需要数据库脚本/seed/验流脚本同步更新。

## 结论

当前 backend 不是“没后端”，而是已经有一条半成品主干：登录、建星球、内容 CRUD、评论点赞、加入下单、会员回写都已经落到了 service 和 Prisma 上。真正拖住一期的不是 schema 数量，而是三件事没有收口：身份可信度、内容访问控制、真实支付。

如果 03 号计划按这个顺序排，04 可以直接围绕现有 service 做增量补齐；如果 03 号改成去补统计表、导出表、积分表，那就会明显偏离当前最短闭环路径。

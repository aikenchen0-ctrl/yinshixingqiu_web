# 知识星球复刻 SOP

## 目标

本项目中的“知识星球”只是小程序中的一个子模块。
第一阶段目标不是一次性做到完全还原，而是先快速做出可运行、结构清晰、方便二次开发的版本。

执行优先级如下：

1. 先把页面结构和核心交互跑起来。
2. 再补齐真实数据结构与接口联调。
3. 最后做细节视觉、性能和体验优化。

## 小程序端执行规则

- 严格使用原生微信小程序技术栈：WXML / WXSS / JS。
- 禁止使用 HTML 标签，必须使用 view、text、image、scroll-view、swiper 等小程序原生标签。
- 禁止使用 window、document、fetch、axios 等非小程序原生能力。
- 所有页面数据更新只能通过 this.setData()。
- 导航必须使用绝对路径。
- tabBar 页面必须使用 wx.switchTab。
- 普通页面使用 wx.navigateTo。
- 图片统一使用网络 URL，不使用 base64。
- 能用原生能力解决的问题，不引入第三方库。

## 体积与目录策略

- 主包必须控制在 2MB 以内。
- 首页、底部 tab、基础导航、公共样式、基础组件优先放主包。
- 知识星球详情、发帖、消息、管理等非首屏关键能力优先考虑分包。
- 禁止为了省事引入大型 npm 依赖。
- 新增代码时，优先考虑是否可以沉淀为公共组件或公共工具。

建议的小程序拆分方向：

- 主包：壳层、首页入口、基础组件、公共样式、登录态基础能力。
- 分包一：知识星球内容浏览。
- 分包二：知识星球互动能力，如发帖、评论、点赞、成员操作。
- 分包三：低频页面，如设置、协议、帮助、活动页。

## Figma 到小程序的工作流

1. 先注册并截图知识星球用户端。
2. 将截图导入 Figma，按真实信息架构快速拼页面。
3. 手动修正布局、层级、间距、字体和视觉样式。
4. 手动标注核心可复用组件，例如：
   - 列表容器
   - 内容卡片
   - tab 切换
   - 用户信息行
   - 操作栏
   - 空状态
   - 顶部导航
5. 再让 Codex 把标注后的页面转成微信小程序代码。
6. 生成代码后，先校验组件复用，再补交互，再联调接口。

## 页面实现原则

- 第一遍复刻优先保证页面骨架、信息层级、滚动体验、交互路径正确。
- 不追求第一版就像素级完全一致。
- 页面中重复出现 2 次及以上的结构，优先考虑抽组件。
- 单个组件保持单一职责，不要把列表、弹窗、数据请求、状态切换全塞进一个组件。
- 样式类名必须语义化，例如：
  - `planet-card`
  - `planet-card__header`
  - `page-section`
  - `tab-bar__item`
- 避免无意义命名，例如：
  - `box1`
  - `left-wrap`
  - `c-item`

## 建议优先沉淀的组件

- 自定义导航栏
- tab 切换组件
- 内容卡片组件
- 列表分页加载组件
- 用户头像与昵称组件
- 空状态组件
- 骨架屏组件
- 底部操作栏组件
- 通用弹窗组件

## 管理端复刻 SOP

管理端以 Web 后台方式复刻，重点不只是“像”，而是结构和数据模型都尽量贴近真实系统。

执行顺序建议如下：

1. 用 Chrome DevTools 查看真实页面结构。
2. 通过 Elements 分析页面布局、表格结构、筛选区、弹窗、抽屉、表单结构。
3. 通过 Network 分析接口地址、请求方式、鉴权方式、分页参数、筛选字段、返回字段。
4. 抽象出管理端的信息架构，例如：
   - 登录
   - 仪表盘
   - 星球管理
   - 内容管理
   - 用户管理
   - 订单或会员管理
   - 配置中心
5. 对照真实结构复刻页面，不足部分可参考 Ant Design Pro、Vue Element Admin 等成熟后台范式补齐。

管理端页面优先复刻这些模式：

- 顶部筛选 + 表格列表
- 详情抽屉 / 详情页
- 新建与编辑表单
- 批量操作
- 状态切换
- 分页、搜索、标签筛选

## 数据结构分析 SOP

数据结构和 UI 复刻可以并行，但接口字段必须尽量基于真实抓包，不要纯猜。

建议按下面顺序提取：

1. 从 Network 中找到核心接口。
2. 记录每个接口的：
   - 路径
   - 方法
   - 请求头
   - 鉴权方式
   - 请求参数
   - 响应字段
3. 建立领域模型草稿，例如：
   - 用户
   - 星球
   - 帖子
   - 评论
   - 话题
   - 订阅 / 会员
   - 订单
   - 消息通知
4. 再回头对照页面，确认每个页面依赖哪些接口和字段。

## 后端技术选型建议

当前阶段优先目标是快速落地、便于 AI 协作、后续易扩展。

默认推荐：

- Node.js
- PostgreSQL
- Redis
- Prisma

推荐原因：

- 对 AI 生成和维护较友好。
- CRUD、鉴权、后台接口、管理端联调通常会更快。
- Prisma 对数据模型和迁移的可读性较好。
- 对 1 万到 5 万用户规模足够用。

可选备选：

- Go + Gin 或 Fiber + PostgreSQL + Redis
- Spring Boot + MySQL + Redis

选型原则：

- 优先开发效率。
- 优先可维护性。
- 优先方便后续平滑演进。
- 不为了“理论最优架构”牺牲当前推进速度。

## Codex 执行偏好

当 Codex 在本仓库中工作时，默认应遵守以下偏好：

- 第一版优先输出能运行、结构清晰、便于二开的代码。
- 小程序端优先做组件复用，不重复造页面块。
- 页面样式优先统一设计 token、间距体系、卡片体系。
- 后端优先按模块划分目录，不把全部逻辑堆在单文件。
- 管理端优先抽公共表格页、筛选栏、弹窗表单模式。
- 涉及真实系统复刻时，优先信任真实 DOM 与 Network 证据，而不是视觉猜测。

## 当前阶段最推荐流程

如果现在开始正式推进，建议顺序如下：

1. 先完成知识星球用户端截图采集。
2. 在 Figma 中完成页面拼装和组件标注。
3. 先让我把用户端子模块转成原生小程序结构。
4. 再同步分析管理端 DOM 和 Network。
5. 产出接口清单和数据模型草稿。
6. 按 Node.js + PostgreSQL + Redis + Prisma 搭第一版后端。
7. 最后做小程序联调、后台联调和体验修正。

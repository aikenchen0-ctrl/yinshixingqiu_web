# 知识星球全栈复刻执行提示词 01

## 你的角色

你是这个仓库的主力全栈工程师，需要继续完善“知识星球”相关能力。请直接在当前仓库工作，而不是只给建议。

## 仓库上下文

- 小程序前端在 `miniprogram`
- 管理端前端在 `admin-web`
- 后端在 `backend`
- 项目根目录有 `AGENTS.md`
- 参考 SOP 在 `docs/rebuild-workflow.md`

## 核心目标

我已经完成了一部分代码，但当前还不够像知识星球，尤其后端管理相关页面和功能存在明显偏差。你需要同时覆盖两个视角：

1. 用户视角
2. 管理视角

参考站点：

- 用户/星球网页入口：`https://wx.zsxq.com/group/28882128518851`
- 管理后台入口：进入该页面后点击右上角“管理后台”

## 强制执行要求

1. 先阅读仓库代码，重点理解 `admin-web` 和 `backend`
2. 对照真实网页做复刻，优先相信真实 DOM、真实 Network、真实交互路径，不要只靠截图猜
3. 每开始一个功能前，先设计测试数据
4. 功能完成后，必须拿测试数据回头验证
5. 必须能读取数据库，核对数据库中的数据结构、种子数据、接口返回是否一致
6. 如果发现某个页面“很不像”，要主动找出差异最大的页面并优先修正
7. 不要只做页面，还要同步检查接口、字段、列表结构、筛选项、状态流转是否贴近真实系统

## 工作方式

请按下面顺序推进：

1. 先盘点当前已完成内容，列出用户端、管理端、后端各自已有模块
2. 找出与真实知识星球差异最大的 3 个页面或功能
3. 给出本轮优先级排序，并说明为什么先做这些
4. 每做一个功能时，先补测试数据方案，再实现，再验证
5. 对 `admin-web` 和 `backend` 联动检查，避免只改一侧
6. 每完成一块后，说明你验证了哪些数据、哪些页面、哪些接口

## 数据与测试要求

每个功能都要提前设计最小可验证测试数据，至少覆盖：

- 正常数据
- 空数据
- 边界数据
- 异常状态数据

示例但不限于：

- 星球基本信息
- 成员列表
- 帖子列表
- 评论列表
- 管理员角色
- 付费/会员状态
- 审核状态
- 分页和筛选结果

## 输出要求

不要泛泛而谈。你需要直接产出以下结果：

1. 代码修改
2. 数据库或测试数据补充
3. 功能验证结果
4. 剩余差距清单

## 约束

- 小程序相关实现必须遵守 `AGENTS.md`
- 小程序只能使用原生微信小程序技术栈
- 管理端允许参考成熟后台范式，但要以真实知识星球后台为主
- 后端优先保持 Node.js + PostgreSQL + Redis + Prisma 方向
- 第一轮以“先有可运行版本”为目标，优先结构清晰、便于二开

- 哪些信息已经确认

- 你是基于什么证据做的合理推断

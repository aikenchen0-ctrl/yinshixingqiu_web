# 知识星球数据闭环执行提示词 03

## 目标

请以“数据结构真实、接口闭环、页面能验证”为最高优先级，继续完善当前仓库中的知识星球模块。不要只做静态页面，也不要只改一个端。

## 当前仓库

- `admin-web`：管理端前端
- `backend`：服务端与数据库相关逻辑
- `miniprogram`：微信小程序端

## 外部参考

- 真实用户视角页面：`https://wx.zsxq.com/group/28882128518851`
- 管理视角：该页面右上角进入“管理后台”

## 关键要求

1. 先从真实页面和真实网络请求中抽取数据模型
2. 再反查当前数据库、Prisma schema、接口定义、前端使用字段
3. 找出哪些页面不像，本质上是因为哪些数据模型没有对齐
4. 优先修复“数据结构不对导致页面不像”的问题
5. 每个功能必须先准备测试数据，再做开发，再拿测试数据验证

## 工作步骤

1. 盘点现有数据库表、核心接口、管理端页面、用户端页面
2. 建立知识星球领域模型草稿，包括但不限于：
   - 用户
   - 星球
   - 成员
   - 帖子
   - 评论
   - 话题
   - 订阅 / 会员
   - 通知
   - 管理员
   - 审核状态
3. 对照真实系统，识别字段缺失、状态缺失、关系缺失
4. 补齐后端模型和接口
5. 回填测试数据
6. 修正 `admin-web` 页面与交互
7. 修正必要的用户视角页面，保证两端数据一致

## 测试数据要求

每个领域对象都要准备可验证样本，不允许只靠一条假数据。至少包含：

- 活跃星球
- 新建星球
- 无帖子星球
- 有置顶帖的星球
- 有审核中内容的星球
- 有被删除/隐藏内容的星球
- 普通成员、付费成员、管理员、过期成员

## 验证要求

每做完一个功能，请明确核对：

- 数据库实际记录
- 接口返回 JSON
- 管理端列表页
- 管理端详情页 / 编辑页
- 用户侧对应展示

## 交付标准

你的输出必须是可继续迭代的工程结果，而不是单纯分析。需要包括：

1. 代码修改
2. 测试数据
3. 验证结论
4. 仍未对齐真实系统的差距清单

## 约束

- 小程序端严格遵守 `AGENTS.md`
- 小程序只用原生能力
- 后端优先保持简单、能跑、可维护
- 不为了理论完美拖慢推进速度

请从“盘点现有数据模型”和“识别最不像的页面”开始。

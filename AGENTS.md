语法隔离：
- 这是一个原生微信小程序项目。
- 禁止使用 HTML 标签（如 div、span），必须使用 WXML（view、text、scroll-view 等）。
- 禁止使用传统的 CSS 媒体查询，优先使用 rpx 进行响应式布局。
- 仅使用原生小程序（WXML / WXSS / JS），能用原生 API 绝不用第三方库。

API 限制：
- 禁止使用 window、document 等 DOM API。
- 所有网络请求必须使用 wx.request，严禁使用 fetch 或 axios。
- 图片使用网络 URL，禁止内嵌 base64。

状态管理规约：
- 修改页面数据必须且只能通过 this.setData()。
- 禁止直接修改 this.data。

路由规则：
- 导航时始终使用绝对路径，以 / 开头，例如 /pages/index/index。
- 如果是 app.json 中定义的 tabBar 页面，必须使用 wx.switchTab。
- 普通页面使用 wx.navigateTo。

体积限制：
- 主包体积必须控制在 2MB 以内。
- 优先使用分包，避免把非首页核心功能堆进主包。
- 禁止引入大型 npm 库，新增依赖前先确认是否能用原生能力实现。

组件与样式规则：
- 优先建立可复用组件体系，能复用组件就不要重复写页面结构。
- 常见能力优先沉淀为单职责组件，例如列表、卡片、tab、弹窗、空状态、导航条。
- 每个组件职责单一，避免一个组件承担过多业务逻辑。
- 样式尽量复用 class，但避免为了抽象而抽象。
- class 命名必须语义化，禁止使用无意义缩写。
- 视觉复刻第一轮以“快速成型、便于二开”为目标，不追求一次性完美像素级还原。

知识星球复刻工作流：
1. 先注册并截图知识星球用户端。
2. 将截图整理进 Figma，先快速拼出页面结构。
3. 在 Figma 中手动修订，并手动标注核心组件，如列表、卡片、tab。
4. 再交给 Codex 转成微信小程序代码，目标产物为 WXML + WXSS + JS。
5. 在微信开发者工具中调试，并重点适配微信登录、获取手机号等微信能力。
6. 知识星球只是整个小程序中的一个子模块，设计和实现时不要把整个项目都绑定成“只有知识星球”。

管理端复刻规则：
1. 管理端优先按 Web 后台来复刻，允许参考 Ant Design Pro、Vue Element Admin 等成熟后台的布局与交互模式。
2. 复刻管理端时，应优先参考真实页面的 DOM 结构和 Network 接口，而不是只靠截图猜测。
3. 如果需要分析管理端，应优先使用 Chrome DevTools 查看 Elements 和 Network。
4. 数据结构分析与界面复刻可以并行推进，但接口、字段、列表结构、筛选条件必须尽量从真实 Network 中还原。

后端搭建偏好：
- 用户规模预估为 1 万到 5 万，总体不需要过度设计。
- 后端目标是快速落地、便于 AI 协作、便于后续平滑优化。
- 默认优先考虑 Node.js + PostgreSQL + Redis + Prisma，这套组合对 AI 协作更友好，通常开发速度更快。
- 如果业务场景明确更适合其他技术栈，也可以选择 Go（Gin / Fiber）+ PostgreSQL + Redis，或 Spring Boot + MySQL + Redis。
- 技术选型优先级：开发效率、可维护性、后续扩展性，优先于“理论上最强”的架构。

执行原则：
- 第一遍复刻以“先有可运行版本”为目标，优先交付能跑、结构清晰、方便二开的代码。
- 第二遍再逐步补齐交互细节、接口联调、性能优化和体验优化。
- 做方案判断时，优先选择对当前阶段更快、更稳、更易维护的实现方式。
- 涉及知识星球复刻、管理端复刻、后端规划时，优先同时参考 docs/rebuild-workflow.md 中的执行 SOP。
